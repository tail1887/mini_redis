# 05. 스터디 가이드

## 1) 학습 목표

- 0~5단계 로드맵을 기능/리스크/완료 조건 관점으로 설명한다.
- API/아키텍처/테스트 문서를 연결해서 이해한다.
- 4인 협업 역할에 맞춰 발표/리뷰/회고를 수행한다.

## 2) 권장 학습 운영(팀 4인)

- 단계 시작 전 30분: 목표/리스크/테스트 포인트 정렬
- 단계 종료 후 20분: 결과 회고(성공 기준 충족 여부 확인)
- 산출물: 단계 요약 1페이지 + 다음 단계 질문 3개

## 3) 0~5단계 학습 로드맵

| 단계 | 핵심 학습 주제 | 성공 기준 | 리스크 | 완료 조건 |
| --- | --- | --- | --- | --- |
| 0 | 공통 용어, 문서 구조, 규칙 | 팀 공통 정의 사용 | 용어 혼선 | 용어/규칙 합의 |
| 1 | KV 명령과 오류 처리 | 입력/출력 계약 설명 가능 | 정상 흐름만 이해 | 명령별 예시 3개 정리 |
| 2 | TTL/만료 경계 | `-2/-1/>=0` 규칙 설명 가능 | 시간 로직 오해 | 경계 시나리오 노트 작성 |
| 3 | prefix 무효화/운영 지표 | 무효화 위험과 보호 규칙 설명 | 과삭제 인식 부족 | 운영 체크포인트 합의 |
| 4 | 부하 결과 해석/릴리스 판단 | 성능 지표로 의사결정 가능 | 수치 해석 부족 | 최종 리포트 리뷰 완료 |
| 5 | 동시성/내구성 복구 | 레이스/복구 흐름 설명 가능 | 영속화 오해 | 복구 데모와 운영 기준 정리 |

## 4) 4인 역할 분담(학습 관점)

| 역할 | 책임 |
| --- | --- |
| PM | 세션 아젠다, 일정, 회고 기록 |
| TL | 기술 주제 정리, 설계 질의 대응 |
| DEV | 구현 관점 사례 공유, 트러블슈팅 |
| QA | 테스트 관점 설명, 게이트 해석 |

## 5) 테스트 분류 학습 포인트

- Unit: 함수/모듈 단위 규칙 검증
- Integration: API부터 저장소까지 흐름 검증
- Failure: 잘못된 입력/경계/오류 처리 검증
- Load: 처리량/지연/오류율 기반 성능 검증

## 6) 세션 체크리스트

- 오늘 단계 성공 기준을 먼저 읽었는가?
- 리스크 대응안을 최소 1개 도출했는가?
- Unit/Integration/Failure/Load 관점으로 설명했는가?
- 문서 변경 필요 항목을 명시했는가?

## 7) 발표용 스토리라인 (문서 순서 기반)

발표는 아래 순서를 고정하면 흐름이 안정적이다.

1. `docs/01`: 왜 이 프로젝트를 단계형으로 설계했는가(문제/범위)
2. `docs/02`: API/Service/Store/Observability 경계를 왜 나눴는가
3. `docs/03`: 단계별 계약(엔드포인트, 입력 규칙, 에러 코드)
4. `docs/04`: 1~5단계 확장 방식과 운영 대책
5. `docs/06`: 검증 방식(Unit/Integration/Failure/Load)과 성능 게이트

핵심 메시지 템플릿:
- "기능 추가"보다 "리스크 통제 + 검증 완료"를 우선했다.
- 단계별 게이트를 통과해야 다음 단계로 넘어간다.
- 성능과 안정성은 감이 아니라 지표(req/s, p95, error rate)로 판단한다.

## 8) 3~4분 발표 스크립트 템플릿

### 8-1) 도입 (30초)

"저희는 Redis 핵심 기능을 한 번에 구현하지 않고, 0~5단계로 나눠 확장했습니다.
핵심 목적은 속도보다 안정성입니다. 문서 계약, 계층 분리, 테스트 게이트를 먼저 고정하고 구현했습니다."

### 8-2) 본문 1: Phase 1~2 (90초)

"1단계는 KV 기본기입니다. 키를 `<prefix>:<name>` 형식으로 강제해 입력 오류를 초기에 차단했습니다.
2단계는 TTL입니다. `-2/-1/N` 규칙으로 상태 해석을 고정했고, 경계 시간 테스트로 만료 오동작을 줄였습니다."

### 8-3) 본문 2: Phase 3~5 (90초)

"3단계는 운영성입니다. `invalidate-prefix`를 넣되 과삭제/과소삭제를 막는 보호 로직과 장애 테스트를 같이 적용했습니다.
4단계는 통합 회귀와 부하 검증으로 성능 기준을 확인했습니다.
5단계에서는 동시성 보호와 AOF/스냅샷 복구를 넣어 재시작 후 데이터 복구까지 검증했습니다."

### 8-4) 마무리 (20초)

"결과적으로 저희가 얻은 것은 기능 목록이 아니라, 확장해도 무너지지 않는 개발 방식입니다.
새 기능이 들어와도 같은 문서-테스트-릴리스 루프로 빠르게 검증할 수 있습니다."

## 9) 장표 구성 예시 (6장)

1. 프로젝트 목표와 0~4단계 로드맵
2. 아키텍처 계층(API/Service/Store/Observability)
3. API 계약(핵심 엔드포인트 + 입력/에러 규칙)
4. Phase 1~2 구현/검증 포인트
5. Phase 3~4 운영/성능/배포 포인트
6. 결론(성과 + 다음 개선 과제)

## 10) 예상 질문/답변 템플릿

Q1. "동시성 이슈 대책이 있나요?"
- A. store 단위 락(`threading.RLock`)으로 프로세스 내부 동시성 레이스를 방지했다.
- A. 병렬 `set/get/expire/persist` 테스트로 상태 일관성을 검증했다.
- A. 멀티 인스턴스 원자성은 후속 확장 과제로 분리했다.

Q2. "서버 다운 대책이 충분한가요?"
- A. 운영 중 자동 페일오버는 이번 범위에서 제외했다.
- A. 대신 AOF + 스냅샷 복구를 넣어 재시작 시 데이터 복구 경로를 확보했다.
- A. 배포 직후 헬스체크 재시도와 롤백 절차도 유지해 운영 리스크를 낮췄다.

Q3. "왜 이렇게 문서가 많나요?"
- A. 팀 협업에서 구현보다 먼저 합의가 깨지기 쉽기 때문이다.
- A. 문서-코드-테스트를 함께 관리해야 단계별 게이트를 객관적으로 통과시킬 수 있다.
