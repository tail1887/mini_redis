# 01. 제품 기획서

## 1) 프로젝트 정의

- 프로젝트명: `mini_redis`
- 목표: Redis 핵심 개념을 0~5단계로 구현하며, 문서-구현-테스트 정합성을 갖춘 팀 개발 방식을 정착한다.
- 개발 언어/플랫폼: `Python + FastAPI + redis-py`
- 팀 구성: 4인(비전공자 기준 단순 4분할)

## 2) 범위

### 포함(MVP)
- 핵심 KV, TTL, prefix 무효화, 성능 검증, 동시성/내구성 보강
- 단계별 성공 기준/리스크/완료 조건 기반 운영
- Unit/Integration/Failure/Load 테스트 체계 도입

### 제외(이번 스프린트)
- Redis 프로토콜 완전 호환
- 클러스터/샤딩/복제 자동 장애복구
- 프로덕션급 보안/관측 스택

## 3) 0~5단계 로드맵

| 단계 | 목표 | 성공 기준 | 주요 리스크 | 완료 조건 |
| --- | --- | --- | --- | --- |
| 0 | Interface First 초기 틀 AI 생성 | API 초안/폴더/테스트 템플릿 자동 생성 | AI 초안 품질 편차 | 팀이 초안 검토 후 수정 목록 확정 |
| 1 | `SET/GET/DEL/EXISTS` 구축 (AI 틀 기반) | 정상/오류 응답이 API 계약과 일치 | 키 검증 불일치 | 핵심 명령 Unit/Integration/Failure 통과 |
| 2 | `EXPIRE/TTL/PERSIST` 도입 (AI 틀 기반) | TTL 규칙(`-2/-1/>=0`) 일관 적용 | 시간 경계 플래키 테스트 | TTL 경계 케이스 포함 테스트 통과 |
| 3 | prefix 무효화 + 운영 지표 (AI 틀 기반) | 의도한 범위만 삭제, 지표 확인 가능 | 과삭제/과소삭제 | 무효화 통합·실패 시나리오 통과 |
| 4 | 통합 회귀 + 부하 검증 (AI 틀 기반) | 합의 성능 기준(req/s, p95, error rate) 충족 | 기능 증가에 따른 성능 저하 | Load 리포트 승인, 릴리스 체크리스트 완료 |
| 5 | 동시성/내구성 안정화 (AI 틀 기반) | 경합 상황 무결성 유지 + 재시작 후 데이터 복구 | 락 경합으로 인한 지연, 파일 손상 | 동시성/복구 테스트 통과, 운영 가이드 승인 |

## 4) 4인 협업 역할 분담(단순 버전)

| 파트 | 담당 범위 | 산출물 |
| --- | --- | --- |
| 파트 1 | API/입력 검증 | 엔드포인트 동작, 요청/응답 예시 |
| 파트 2 | 핵심 KV 구현 | set/get/del/exists 로직 |
| 파트 3 | TTL/무효화 | expire/ttl/persist/invalidate 로직 |
| 파트 4 | 테스트/문서 | Unit/Integration/Failure/Load 결과와 문서 갱신 |

## 5) 단계 운영 원칙

- 단계 0은 팀이 직접 작성하지 않고 AI 초안으로 시작한다.
- 단계 1~5도 각 단계 시작 전에 AI로 해당 단계 전용 초기틀을 생성한다.
- 단계 진입 전: 목표/성공 기준/리스크/완료 조건이 문서에 명시되어야 한다.
- 단계 진행 중: 리스크 발생 시 같은 단계 내 대응안(완화/우회/후속)을 기록한다.
- 단계 종료 시: 코드, 테스트 결과, 문서 갱신이 함께 완료되어야 한다.
- 게이트 미통과 시: 다음 단계로 넘어가지 않는다.

### 단계별 AI 초기틀 생성 체크리스트

- API 계약 변경 여부 확인 (`docs/03`)
- AI 프롬프트에 단계 목표/범위/금지사항 명시
- 코드 스켈레톤 + 테스트 템플릿 자동 생성
- 팀 검토 후 수정 항목 확정
- 확정 이후 구현 시작

## 6) 최종 완료 정의(DoD)

- 0~5단계 모두 완료 조건 충족
- 4인 역할별 산출물(결정 로그, PR, 테스트 리포트) 추적 가능
- `docs/01~06` 간 내용 충돌 없음
